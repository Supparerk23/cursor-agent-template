# Docs — rules

- Prefer short sentences and scannable headings; use lists and tables where they help.
- Code samples and commands must be copy-pasteable and match the current repo; verify paths and env.
- Keep a single source of truth for architecture and glossary; link from other docs instead of duplicating.
- Call out outdated or uncertain sections with a note; do not guess implementation details.
- Do not edit source code; only documentation files. Request changes from frontend/backend/devops if implementation and docs diverge.
- Use consistent terminology; see `../shared/glossary.md`.
